---
name: migration-writer
description: Writes new Flyway migrations under backend/src/main/resources/db/migration/ following Picsou's naming, numbering, and entity-mapping conventions. Use when a schema change is needed, or when asked to add a column, table, index, or enum.
tools: Read, Grep, Glob, Write, Edit, Bash
---

Read `docs/conventions/database.md` before writing anything.

Process:

1. **Determine the next version number.** Migration numbering spans branches (`docs/CODING_RULES.md`
   #4) — list migration files on the current branch AND check `git log`/`git branch -a` /
   `git fetch` for the highest `V<n>` on other active branches (especially `main` and any open
   feature branches) before picking `V{next}`. Note that `V8` is intentionally absent by design —
   never reuse it.
2. **Name the file** `V{n}__snake_case_description.sql`.
3. **Write plain SQL** — no Hibernate-generated DDL. Include constraints, indexes, and foreign
   keys explicitly:
   - `GENERATED BY DEFAULT AS IDENTITY` for primary keys (never a sequence Hibernate manages).
   - `NUMERIC(20, 8)` for monetary values, `TIMESTAMPTZ` for timestamps, `DATE` for dates.
   - `ON DELETE CASCADE` on child-table foreign keys unless orphan rows are intentional.
   - Named constraints: `CONSTRAINT uk_{table}_{columns}` / `idx_{table}_{columns}`.
   - New enums via `CREATE TYPE ... AS ENUM (...)` at the top of the file.
4. **Never edit a merged migration** — migrations are append-only.
5. **Pair it with the entity change** if one is needed: Lombok
   `@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder`, `GenerationType.IDENTITY`,
   `@JdbcTypeCode(SqlTypes.NAMED_ENUM)` + `columnDefinition` for Postgres enums, `@Builder.Default`
   matching the SQL `DEFAULT`. Remember `ddl-auto` is `validate` — the entity must match the
   migration exactly or the app fails to start.
6. **Verify** by running the app locally so Flyway applies the migration, or at minimum run
   `mvn test` for any repository test touching the new schema.
7. Update `docs/conventions/database.md`'s migration table with the new entry as part of the same
   change (that table is documentation, not a convention being loosened, so this doesn't trip the
   CODING_RULES.md #0 meta-rule).

Don't guess table/column names from vibes — check existing migrations for the naming pattern used
by adjacent tables before introducing a new one.
